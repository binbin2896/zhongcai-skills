# 问庭.skill · 庭审问题清单设计

> "问"=庭审提问，"庭"=仲裁庭。以问题为刀，剖开案件真相。

## Overview

本Skill在庭前准备阶段（阅卷完成后）生成系统化的庭审问题清单。基于"由表及里、层层递进"原则，按程序问题→实体问题→争议焦点问题的顺序编排，覆盖建工8大类80+典型问题和商品房8大类60+典型问题。

核心定位：🔵 AI根据案件类型生成初始清单 → 🟢 AI补充关联问题 → 🔴 人精选10-15个核心问题并定稿

## When to Use

| 触发信号 | 场景 |
|---------|------|
| "帮我设计庭审问题" | 阅卷完成后，庭前准备阶段 |
| "这个案子庭审要问什么" | 需要针对性设计提问策略 |
| "建工案件庭审清单" | 明确指定建工案件类型 |

不适用场景：案件已开庭（应走 裁笔.skill）/ 未阅卷（应先用 阅卷.skill）

## AI/Human 分工

🔵 AI自动：根据案由选择对应清单模板，填充通用问题
🟢 AI建议：基于案件特殊事实补充关联问题，标注争议焦点
🔴 人决定：精选10-15个核心问题，添加案件特殊问题，调整提问顺序

## Execution Flow

### Step 1：加载案件信息

从阅卷笔录或用户提供的案件概要中提取：案由、争议类型、合同性质。

🔴 CHECKPOINT 🛑 — 案由确认：展示提取结果给用户确认。确认→进入Step 2；纠正→重新提取；信息不足→引导补充。

### Step 2：选择问题模板

AI根据案由选择：

| 案由 | 模板 | 问题总数 |
|------|------|---------|
| 建设工程 | 建工8大类 | 80+典型问题 |
| 商品房买卖 | 商品房8大类 | 60+典型问题 |
| 其他 | 通用型3部分 | 30+基础问题 |

🔴 CHECKPOINT 🛑 — 模板确认：展示所选模板给用户，确认后再生成。确认→进入Step 3；换模板→按指令切换；需定制→暂停自动生成，接收用户补充。

### Step 3：生成定制清单

按"程序→实体→争议焦点"三层编排：

- 第一部分：程序性问题（仲裁协议效力、主体资格、请求明确性）
- 第二部分：实体性问题（合同关系、效力、履行、损失）
- 第三部分：调解问题（意愿、方案）

### Step 4：输出庭审问题清单

🔴 CHECKPOINT 🛑 — 定稿确认：展示完整清单给用户，确认后再输出。

- 用户确认 → 输出最终清单
- 用户要求精简 → AI按指令删减问题
- 用户要求补充 → 从参考库中补充关联问题
- 用户推翻方向 → 退回Step 1重新确认案由

最终输出格式：

```markdown
═══════════════════════════════════════════
庭审问题清单
═══════════════════════════════════════════

📋 案件：[AI自动填充案号] | [AI自动填充案由]

一、程序性问题（开庭时确认 — 由AI生成，人确认是否完整）
─────────────────────────────────────────────
1. [问题内容] → 依据：[法律/合同条款/仲裁规则]
   💡 提示：代理权限必须确认是否特别授权
2. [问题内容] → 依据：[法律/合同条款/仲裁规则]
...

二、实体性问题（庭审调查核心 — 由AI出全集，人精选10-15个）
─────────────────────────────────────────────
🔴 合同效力类
1. [问题] → 🔵 AI填充
2. [问题] → 🔵 AI填充

🟡 履行情况类
1. [问题] → 🔵 AI填充
2. [问题] → 🔵 AI填充

🟢 损失与责任类
1. [问题] → 🔵 AI填充
2. [问题] → 🔵 AI填充

   ⚠️ 约束规则：
   - 三类问题一定要平衡，不能全是效力类没有履行类
   - 每个争议焦点至少对应1-2个问题
   - 问题不重复、不包含答案暗示

三、调解问题（庭前中后三个时机 — 由AI建议三个时机，人决定策略）
─────────────────────────────────────────────
1. 庭前：双方是否有调解意愿？
2. 庭中：是否同意仲裁庭主持调解？
3. 庭后：是否需要给当事人冷静期再跟进？

💡 提示：争议焦点范围越小越好
   输出前须过自检表，全部✅方可交付
```

### 填写规范

| 字段 | 填写人 | 规则 |
|------|--------|------|
| 案号/案由 | 🔵 AI | 从Step 1提取，用户确认 |
| 程序性问题 | 🔵 AI生成 | 必须包含：回避询问、代理权限、反请求期限 |
| 实体性问题的初选 | 🔵 AI生成 | 按模板从80+/60+库中匹配，全部输出 |
| 实体性问题的精选 | 🔴 人决定 | 从AI全集中选10-15个，删掉不相关的 |
| 问题措辞调整 | 🔴 人决定 | 针对案件特殊事实微调 |
| 调解策略 | 🟢 AI建议 + 🔴 人定 | AI给出三个时机的话术建议，人选择

## 生态集成：上下游对接

### 在经纬skill链中的位置

```
阅卷.skill (上游) ──→ 问庭.skill ──→ 裁笔.skill (下游)
     │                     │                    │
     输出：阅卷笔录          输出：庭审问题清单      输出：裁决书初稿
     （含案由/争议焦点）     （精选10-15问）        （含认定事实/说理）
```

### 上游对接（阅卷.skill → 问庭.skill）

| 上游产出 | 本skill消费方式 | 传递要求 |
|---------|---------------|---------|
| 案由 | 自动填充到Step 1，用于选择模板 | 必须有，否则触发F002降级 |
| 争议焦点初步归纳 | 直接映射到实体性问题的🔴🟡🟢分类 | 建议有，无则AI从仲裁请求反推 |
| 合同性质/关键事实 | 用于定制问题措辞 | 可选，有则输出更精准 |

### 下游对接（问庭.skill → 裁笔.skill）

| 本skill产出 | 下游消费方式 | 传递格式 |
|------------|-------------|---------|
| 争议焦点确认 | 裁笔.skill Step 5的仲裁庭意见框架 | 直接复用 |
| 庭审问题 | 裁笔.skill Step 3的证据和事实认定 | 参考使用 |
| 调解结果 | 裁笔.skill的裁决书版本选择（调解成功则用调解版） | 若调解成功则跳过裁笔 |

### 与其他skill的边界

| 不要用问庭.skill的场景 | 应用的skill |
|----------------------|------------|
| 案卷刚到，还没看过 | 阅卷.skill（先做阅卷笔录） |
| 庭审已结束，需要写裁决书 | 裁笔.skill |
| 裁决书写完了，要质检 | 校核.skill |

## 异常处理与降级路径

| 故障码 | 异常场景 | 触发条件 | 一线修复 | 仍失败兜底 |
|-------|---------|---------|---------|-----------|
| F001 | 用户未阅卷，直接要求出庭问题 | 用户无法提供案由/争议焦点 | 退回推荐先走 阅卷.skill | 用户坚持 → 降级通用模板，标注"未经阅卷" |
| F002 | 案由信息不足，无法匹配模板 | 用户只说了"帮我看个案子" | 追问3个维度：案件类型/争议核心/当前阶段 | 仍不提供 → 通用模板，标注"信息有限" |
| F003 | CHECKPOINT 用户说方向不对 | Step 2/3 用户指出案由判断错误 | 退回 Step 1 重新加载案件信息 | 无法说明正确方向 → 通用模板，用户边看边改 |
| F004 | 用户对生成的清单不满意 | Step 4 预览时要求大量调整 | 按具体要求精简/补充/修正 | 仍不满意 → 输出原始版，标注"待手动编辑" |
| F005 | 用户中途说"算了我自己来" | 任意环节用户中断 | 保存当前进度，告知后续可继续 | — |

## 反模式与黑名单

| # | 反模式 | 为什么不行 | 正确做法 |
|---|--------|-----------|---------|
| 1 | 问题太多庭上问不完 | AI生成了80+个问题，但开庭时间通常只有2-4小时，全问不现实 | AI出完整清单→人精选10-15个核心，按程序→实体→调解顺序排 |
| 2 | 争议焦点归纳过宽 | 直接把仲裁请求（"要求支付工程款"）当焦点，范围太大无法深入 | 按三分法拆为事实/法律/程序子焦点，范围越小越好 |
| 3 | 遗漏程序性环节 | 上来就问实体问题，仲裁协议效力和代理权限没确认 | 无论案件多简单，开庭第一件事必须是：协议效力+回避+代理权限 |
| 4 | 过度依赖AI不做精选 | AI生成了清单就照单全收，80个问题全部打印带上庭 | AI负责出全集，人负责做减法——精选10-15个，省下时间追问细节 |
| 5 | 把调解问题当走过场 | 庭审结束时随口问一句"愿不愿调解"，当事人说"不"就结束 | 调解在庭审前、中、后三个时机分别询问，庭后给当事人冷静期再跟进 |

## 使用示例

### 输入

> 案由：建设工程施工合同纠纷。合同为固定总价，工程已竣工验收。申请人主张被申请人拖欠工程款1250万元，被申请人以工程质量不合格提起反诉。争议焦点：①已完工程量如何认定 ②质量缺陷责任归属 ③优先受偿权是否过期。

### ❌ 坏例（未使用本skill）

直接输出几个笼统的问题，缺乏层次和针对性：

```
1. 你们合同怎么签的？
2. 工程款付了多少？
3. 工程质量有没有问题？
```

**问题：** 没有程序性环节确认、没有按争议焦点逐层深入、无法覆盖80+建工专业问题、调解环节缺失。

### ✅ 好例（使用本skill）

**一、程序性问题**

1. 双方当事人对仲裁庭的组成是否有异议？是否申请回避？
2. 申请人代理人代理权限？是否特别授权？（一般代理不能代为承认或放弃仲裁请求）
3. 被申请人反诉请求是否明确？是否在答辩期内提出？

**二、实体性问题**

🔴 **合同效力类**
4. 案涉工程是否属于必须招标的项目？固定总价合同中约定的包干范围包括哪些内容？
5. 施工过程中是否存在设计变更或签证？如有，固定总价是否调整？

🟡 **履行情况类**
6. 案涉工程竣工验收日期是什么时候？双方是否签署了竣工结算文件？
7. 申请人主张的1250万元是如何计算得出的？是否有结算书或鉴定意见支撑？
8. 被申请人主张的质量缺陷具体有哪些？是竣工验收前已存在的还是竣工后发现的新问题？

🟢 **损失与责任类**
9. 质量缺陷的责任归属——是施工原因、设计原因还是材料原因？如是材料原因，由谁提供？
10. 申请人主张优先受偿权的起算时间是否在18个月期限内？（《民法典》第807条）
11. 工程款利息的起算日是哪一天？合同对逾期付款违约金是否有约定？

**三、调解问题**
12. 双方是否有调解意愿？如有，是否同意庭后调解？

### 对比要点

| 维度 | 坏例 | 好例 |
|------|------|------|
| 程序确认 | 无 | 代理权限、回避、反请求期限——全部覆盖 |
| 层次 | 平铺直叙 | 程序→效力→履行→责任，逐层深入 |
| 专业深度 | 泛泛而问 | 引用固定总价包干范围、18个月期限、民法典807条 |
| 调解 | 无 | 独立一节，给出明确时机 |

## 自洽性与质量自检

### 输出前自检表

逐项确认后方可输出最终清单：

| 检查项 | 问题 | 通过标准 |
|--------|------|---------|
| 🎯 核 | 问题清单是否覆盖了全部争议焦点？ | 用户确认的每个争议焦点至少有1个对应问题 |
| 📋 全 | 程序→实体→调解三层是否完整？ | 三层都有内容，无遗漏 |
| 🎯 精 | 问题数量是否控制在合理范围？ | 10-15个核心问题，不超过20个 |
| 🔗 序 | 问题顺序是否符合"由表及里"原则？ | 先程序、再实体、最后调解，实体中先效力后履行再责任 |
| ⚠️ 疑 | 是否存在可能遗漏的关键问题？ | 涉及鉴定/时效/管辖权等程序性前提的，已包含 |
| ✅ 用 | 清单在庭上是否可执行？ | 每个问题都是一句能问完的，不含复合多重提问 |

🛑 **强制回退规则**

- **任一❌ → 退回对应步骤补齐**，修正后再检
- **连续2次❌同一检查项** → 建议用户先走 阅卷.skill 完成案卷梳理
- **CHECKPOINT 用户中止** → 保存当前进度，按用户指明方向重走
- **全部✅ → 输出最终清单**
