# 经纬.skill · 仲裁执业技能体系入口

> 《左传》"经纬天地曰文"——案件经纬万端，仲裁员持经纬以度之。

## Overview

经纬是仲裁skill链的**入口编排（meta-skill）**。它不执行具体的校核或阅卷操作，而是根据当前案件所处的阶段，判断应该加载哪个子skill，并完成上下文传递。

## Skill Chain Architecture

```
经纬.skill（入口编排）
├─ 阅卷.skill（庭前准备 → 三层五要素阅卷）
├─ 问庭.skill（庭前准备 → 庭审问题清单）
├─ 裁笔.skill（裁决书草拟 → 结构+案例引用）
├─ 校核.skill（裁决书质检 → 三十项校核+风险审查）
├─ 防讼.skill（司法审查风险 → Premortem预演+撤裁防范）
└─ 案归.skill（结案归档 → 复盘+经验入库）
```

## When to Use

| 触发信号 | 行为 |
|---------|------|
| "帮我处理这个仲裁案件" | 自动判断当前阶段，加载对应子skill |
| "走一遍仲裁流程" | 按顺序执行全流程（阅卷→问庭→裁笔→校核→案归） |
| "这个案子应该从哪一步开始？" | 评估案件成熟度，推荐起点 |

## Stage Detection

AI根据用户提供的材料类型和上下文自动判断当前阶段：

| 用户提供了 | 判断为 | 推荐加载 |
|-----------|--------|---------|
| 仲裁申请书+证据材料 | 庭前阶段 | 阅卷.skill |
| 已开庭，需要写裁决书 | 裁决阶段 | 裁笔.skill |
| 裁决书草稿已完成 | 校核阶段 | 校核.skill |
| 案件已审结 | 归档阶段 | 案归.skill |
| 不确定阶段 | 询问用户 | — |

#### 🔴 CHECKPOINT 🛑 阶段确认
在路由到子skill之前，必须向用户展示阶段判断结论并确认。格式："根据您提供的材料，当前处于【庭前/裁决/质检/归档】阶段 → 推荐加载【子skill名】。确认？"用户回复"确认"后再路由。如用户指出阶段判断错误，退回重新检测。

#### 🟡 自动路由决策提示
- 全流程："收到，按阅卷→问庭→裁笔→校核→防讼→案归顺序走完全流程。是否从第一步开始？"
- 中途接入："您已处于【X】阶段，已跳过阅卷/问庭，从【Y】开始。需要回顾已完成步骤吗？"

| Skill | 阶段 | 输入 | 输出 | 状态 |
|-------|------|------|------|------|
| 阅卷 | 庭前准备 | 案卷材料 | 阅卷笔录 | 🟢 可用 |
| 问庭 | 庭前准备 | 阅卷笔录 | 庭审问题清单 | 🟢 可用 |
| 裁笔 | 裁决草拟 | 庭审记录+阅卷笔录 | 裁决书初稿 | 🟢 可用 |
| 校核 | 裁决质检 | 裁决书草稿 | 校核报告 | 🟢 可用 |
| 防讼 | 裁决质检 | 裁决书草稿 | 防讼报告 | 🟢 可用 |
| 案归 | 结案归档 | 全案材料 | 复盘卡 | 🟢 可用 |

#### 🔴 CHECKPOINT 🛑 子skill产出确认
子skill执行完成后，检查产出物是否通过其Verification Checklist。未通过则提示用户：'【产出物名】尚未通过校核，请先完成质检后再进入下一阶段。'已通过则展示摘要并询问：'继续进入下一阶段【下一skill名】？'

#### 🔴 CHECKPOINT 🛑 全流程出口确认
全流程路由完成后（阅卷→案归），输出流程日志摘要：共执行N个步骤、产出物清单、耗时预估。确认无误后回复"完成"关闭。如需批量处理下一个案件，回复"下一个案件"重置。

## 反模式与黑名单（Anti-Patterns）

| # | 反模式 | 为什么有害 | 正确做法 |
|---|--------|-----------|---------|
| 1 | **把经纬当普通skill直接调用子流程**：用户说"帮我阅卷"，经纬却执行阅卷.skill的全部流程 | 经纬的职责是路由，不是执行。越俎代庖混淆了编排层和执行层的边界 | 经纬只做三件事：判断阶段→推荐子skill→传递上下文。具体阅卷/裁决操作由子skill独立完成 |
| 2 | **跳过阶段检测直接路由**：用户提供了材料但不做阶段判断，直接推荐裁笔.skill | 跳过检测可能推荐错误阶段，用户跟着错误的路线走浪费大量时间 | 即使肉眼可见是裁决阶段，也必须显式确认后再路由——一句"我看您已经写完了裁决书草稿，接下来走校核+防讼质检流程，可以吗？" |
| 3 | **全流程模式不分案由**：建工案件和商品房案件都走同样的全流程路由 | 建工案件可能需要造价鉴定专题（阅卷时特别标注），商品房案件可能只需快审通道；一刀切浪费资源 | 在执行全流程路由前，先询问案由类型，根据案由推荐对应的路径模板（如：建工=阅卷+问庭+裁笔+校核+防讼+案归；简易=阅卷+裁笔+案归） |
| 4 | **子skill产出物不验证即传递**：阅卷笔录填了一半就直接完成，问庭.skill拿到半成品清单 | 半成品传到下游，整个链条的产出质量被第一个环节的完成度绑架 | 每个子skill完成后必须通过其Verification Checklist，产出物确认完整后再传递到下一阶段 |
| 5 | **路由日志不记录**：经纬执行了多次路由但没有任何日志记录 | 用户无法回溯案子走到哪一步了，子skill重跑时无法加载中间状态 | 每次路由写入结构化日志：时间戳→用户诉求→阶段判断→推荐skill→产出物状态→下一阶段推荐 |

## 反例与参考（Worked Example）

### 完整交互示例：阶段识别→路由→产出确认

**输入（用户）：** 帮我处理这个仲裁案件——收到仲裁申请书、答辩书和证据材料，还没开庭。

**❌ 坏例（跳过确认直接路由）：**
AI：已识别为庭前阶段→加载阅卷.skill→正在执行三层五要素阅卷...

→ 问题：没有向用户展示阶段判断结论，擅自进入子skill执行。用户可能想说其实已经开过一次庭了。

**✅ 好例（三步标准流程）：**
AI：根据您提供的材料（申请书+答辩书+证据），当前处于【庭前阶段】→推荐加载【阅卷.skill】进行三层五要素阅卷。是否确认？
用户：确认
AI：→已加载阅卷.skill。完成阅卷后，问庭.skill将自动推荐。
阅卷完成后：
AI：阅卷笔录已通过校核。继续进入问庭.skill生成庭审问题清单？
用户：好
AI：→加载问庭.skill...

**对比要点：**
| 对比项 | 坏例 | 好例 |
|--------|------|------|
| 阶段判断 | 暗箱操作，不展示推理过程 | 显式展示判断结论，用户可纠偏 |
| 用户参与 | 零交互，直接执行 | 每次路由前确认，用户有权否决 |
| 错误容忍 | 用户只能被动跟随 | 用户可纠正阶段判断，退回重检 |
| 流程可见性 | 黑盒，无上下文 | 透明，每步展示当前阶段+下一步推荐 |

## 失败模式编码（Failure Mode Encoding）

| 故障码 | 症状 | 触发条件 | 一线修复 | 仍失败兜底 |
|--------|------|---------|---------|-----------|
| F001 | **阶段检测模糊** | 用户提供材料不完整（仅说"帮我处理这个案子"无附件），Stage Detection无法判断 | AI追问2-3个关键问题引导用户明确阶段：案卷到哪一步了？裁决书写了没？开庭了没？ | 退化为"人工选择模式"：展示6个sub-skill列表供用户自选，并在所选skill上方标注⚠️阶段未确认 |
| F002 | **多阶段材料冲突** | 用户同时提供了申请书（庭前）和裁决书草稿（质检），两个不同阶段的判断同时触发 | AI列出两阶段判断的依据，询问用户优先处理哪一个 | 按时间顺序优先：先走完阅卷→问庭，再走校核→防讼，并在路由日志中标注"两阶段暂存" |
| F003 | **子skill加载失败** | 路由至子skill时，skill_view(name)返回错误（skill不存在/路径变更） | AI展示可用skill列表（skills_list），检查是否存在仲裁系列skill，用完整路径重试 | 退化为"人工操作建议"：告诉用户手动加载skill的替代路径，并提供skill的核心指令摘要 |
| F004 | **用户中途切换诉求** | 正在执行校核.skill，用户说"先别校核了，帮我看看这个证据的新材料" | AI暂停当前子skill，切换到阅卷.skill处理新材料，并在日志中标记校核任务为"挂起" | 挂起超过24小时未恢复→自动生成校核进度简报推送用户，由用户决定继续还是放弃 |
| F005 | **全流程路由中断** | 用户走全流程模式，但在某子skill步骤卡住超时 | AI自动检查卡住步骤的输入是否完整，提示缺失前置条件 | 标记为流程断裂，输出已完成步骤汇总，建议用户完成后手动触发下一阶段 |

## 自洽性与质量自检（Verification & Self-Check）

### 路由前自检表（5维）

| # | 检查项 | 标准 | 不通过则 |
|---|--------|------|---------|
| 1 | **阶段判断可追溯** | 阶段判断结论有明确的材料依据（如"根据申请书+证据→庭前"），而非凭猜测 | 追加追问用户至少2个材料特征，直到判断置信度>80% |
| 2 | **子skill可用性** | 目标子skill的metadata中status为🟢可用 | 标记为⚠️降级可用，提示用户该skill可能尚未优化 |
| 3 | **上下游依赖完整** | 目标子skill的输入条件已满足（如裁笔需要阅卷笔录和庭审记录） | 先加载前置skill补齐输入，再路由到目标子skill |
| 4 | **上下文传递完整** | 子skill所需的案号/案由/阶段信息已随路由传递 | 补充缺失上下文后再加载，避免子skill从头追问 |
| 5 | **日志写入确认** | 每次路由操作已完成结构化日志记录 | 补写日志后再进行下一次路由 |

### 🛑 强制回退规则

| 条件 | 动作 |
|------|------|
| 用户指出阶段判断错误 | 退回阶段检测，AI重新分析材料后修正判断 |
| 自检表连续2项不通过 | 退化为手动选择模式：展示6个子skill列表+当前已知信息，用户自选 |
| 同一案件的回退次数>3次 | 标记为该案件不适合自动化路由→全程手动模式，AI仅提供建议 |
