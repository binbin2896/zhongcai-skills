# 防讼.skill · 司法审查风险预判与撤裁防范

> 《论语》"必也使无讼乎"——最好的应诉，是不必应诉。

## Overview

本Skill在裁决书出具前（或双方对抗激烈时）执行**反向压力测试**：站在对方律师的立场，逐项扫描可能被用于申请撤销/不予执行的攻击路径。与校核.skill的"自查"不同，防讼做的是**"预演"**——先假设裁决已被挑战，反向推导漏洞。

核心定位：🔵 AI模拟对方攻击路径 → 🟢 AI评估攻击成功率 → 🔴 人决定是否加固

## When to Use

| 触发信号 | 场景 |
|---------|------|
| "看看有没有撤裁风险" | 裁决书草稿完成，需要从对方角度做风险评估 |
| "对方可能会从哪些角度攻击" | 双方对抗激烈，预判司法审查风险 |
| "走一遍防讼流程" | 与校核.skill配合，形成双层质检 |

不适用场景：案件标的小/事实简单无争议（过度评估）/ 裁决书已送达（来不及了）

## AI/Human 分工

🔵 AI自动：逐项扫描5大撤裁事由，标注每个事由的攻击可能性和事实支撑
🟢 AI建议：针对高风险项给出加固建议（补充记录/强化说理/程序补正）
🔴 人决定：确认风险等级，决定是否修改裁决书或补充程序记录

## Execution Flow

### Step 1：加载裁决书与案件信息

从裁决书草稿+案卷材料中提取关键信息。

🔴 CHECKPOINT 🛑 — 输入确认：展示案件概况给用户确认。确认→进入Step 2；裁决书版本不是最新→先更新；信息不足→追问补充。

### Step 2：撤裁事由逐项压力测试（Premortem）

| 事由 | AI模拟攻击路径 | 事实支撑 | 攻击成功率 |
|------|--------------|---------|-----------|
| **违反法定程序** | 对方可能主张[具体程序瑕疵] | [裁决书中相关记录] | 🔴🟡🟢 |
| **超裁** | 对方可能主张[超范围请求] | [仲裁协议范围vs裁决范围] | 🔴🟡🟢 |
| **隐瞒足以影响公正裁决的证据** | 对方可能主张[未提交的关键证据] | [庭审中是否询问过] | 🔴🟡🟢 |
| **伪造证据** | 对方可能主张[某证据系伪造] | [证据原件核对情况] | 🔴🟡🟢 |
| **枉法裁决** | 对方可能主张[程序严重不公] | [庭审记录完整度] | 🔴🟡🟢 |

### Step 3：深度挖掘（针对高风险项）

对攻击成功率评估为🟡或🔴的事由，做5-Whys深挖：

```
为什么对方可能以[程序瑕疵]为由申请撤裁？
  ↓ 因为 [具体环节] 的记录可能不完整
  ↓ 为什么记录不完整？→ [原因]
  ↓ 可以怎么弥补？→ [加固措施]
```

### Step 4：加固建议

| 风险等级 | 攻击路径 | 加固建议 | 紧急程度 |
|---------|---------|---------|---------|
| 🔴 高 | 送达程序记录不完整 | 补充送达回证+电话记录 | 出裁决前必须补 |
| 🟡 中 | 合议记录缺失 | 补充合议笔录 | 建议补 |
| 🟢 低 | 当事人主张"隐瞒证据" | 裁决书中说明"法庭已询问双方是否还有证据需提交" | 文字上强化即可 |

### Step 5：输出防讼报告

```markdown
🔴 高风险攻击路径（__条）
| 攻击路径 | 事由 | 漏洞描述 | 加固建议 | 跟踪状态 |
|---------|------|---------|---------|---------|
| 1 | | | | □待处理 |

| 攻击路径 | 事由 | 加固建议 |
|---------|------|---------|

🟢 低风险（__条，已加固不需额外动作）

📊 综合风险评级：[低/中/高] 🔴 人确认

```

### 填写规范

| 字段 | 填写人 | 规则 |
|------|--------|------|
| 攻击路径 | 🔵 AI模拟 | 每条必须有事实支撑，不能凭空假设 |
| 攻击成功率 | 🔵 AI评估 + 🔴 人确认 | 没有事实支撑的标🟢低；人与AI不一致以人为准 |
| 加固建议 | 🟢 AI建议 | 必须是可执行动作（补充记录/强化说理/程序补正） |
| 跟踪状态 | 🔴 人更新 | 出裁决前必须确认所有🔴项状态为"已完成"或"已知风险决定不修" |
| 综合评级 | 🟢 AI建议 + 🔴 人定 | AI根据5大事由综合评估，人确认最终等级 |

## 使用示例

### 输入

> 裁决书草稿：建工合同纠纷。申请人主张工程款1250万元，被申请人以质量不合格反诉。已竣工验收。争议焦点：工程量认定、质量缺陷责任、优先受偿权。

### ❌ 坏例（未使用本skill）

```
这个裁决书写得还行，应该不会被撤销。
```

**问题：** 没有逐事由分析、没有攻击路径模拟、没有加固建议、无法量化风险。

### ✅ 好例（使用本skill）

逐事由 premortem，输出分级防讼报告（见模板）。

### 对比要点

| 维度 | 坏例 | 好例 |
|------|------|------|
| 视角 | 自我感觉 | 模拟对方律师 |
| 方法 | 直觉判断 | Premortem预验尸 |
| 输出 | 一句话 | 5事由+攻击路径+加固+追踪 |
| 风险量化 | 无 | 高/中/低+评级 |

## 生态集成：上下游对接

### 在经纬skill链中的位置

```
裁笔.skill → 校核.skill → 防讼.skill（最后一道关）
                               │
                             出裁决
```

### 上下游对接

| 上游（校核）产出 | 本skill消费 | 本skill产出 | 下游 |
|----------------|------------|------------|------|
| 校核报告中的🔴项 | Step 3深度挖掘的重点 | 防讼报告 | 出裁决前签字确认 |

### 边界

| 不要用防讼.skill的场景 | 应用 |
|----------------------|------|
| 裁决书还没写完 | 裁笔.skill |
| 需要校核裁决书质量 | 校核.skill（先校核） |

## 反模式与黑名单

| # | 反模式 | 为什么不行 | 正确做法 |
|---|--------|-----------|---------|
| 1 | 与校核.skill混用 | 校核查"写得好不好"，防讼查"对方从哪攻击"。混用后要么重复劳动要么遗漏视角 | 先校核后防讼，校核产出→防讼输入，两套流程不合并 |
| 2 | 过度评估—所有事由全标🔴 | 5大事由全高说明裁决书有根本性问题，不应出裁决。防讼的目的是找2-3条真实攻击路径，不是制造焦虑 | 标注攻击成功率时必须有事实支撑，没有事实支撑的认定为🟢低 |
| 3 | 忽视隐性程序风险 | 程序性瑕疵（送达/审限/合议记录）比实体错误更容易成为撤裁理由 | 优先检查程序类攻击路径（陶新琴数据：违反法定程序是最高频事由） |
| 4 | Premortem变成普通检查 | 预验尸的精髓是"假设已失败倒推原因"，但做成了普通清单检查 | 每次开始前强制假设"这份裁决已被申请撤销"，以这个心态走完全流程 |
| 5 | 加固建议完成后不跟踪 | 报告出了、建议给了，但没有确认是否真的执行了加固 | 加固建议必须标注"已完成/待处理/不处理"，出裁决前再确认一次 |

## 异常处理与降级路径

| 故障码 | 异常场景 | 触发条件 | 一线修复 | 仍失败兜底 |
|-------|---------|---------|---------|-----------|
| F001 | 裁决书未提供 | 用户说"看看撤裁风险"但没给裁决书 | 要求提供裁决书草稿或Feishu链接 | 用户无法提供→基于已有描述做初步判断，标注"信息有限" |
| F002 | 案件事实过于简单 | 标的小/事实清楚无争议，防讼评估过度 | 告知用户"本案撤裁风险极低，走完标准流程约5分钟" | 用户坚持→快速走完流程，报告标注"低风险案件" |
| F003 | 用户不认同攻击路径判断 | 用户说"对方不会从这个角度攻击" | 记录用户的不同意见，标注"AI建议vs人判断" | 保留双记录，以人判断优先 |
| F004 | 加固建议无法执行 | 用户说"这个来不及补了" | 降级为"风险告知"：标注为已知风险，出裁决前签字确认已知 | 用户确认已知风险仍选择不修→报告标注"已知风险未修复" |
| F005 | 用户中途说"算了" | 任意环节用户中断 | 保存已完成的攻击路径分析，告知可后续继续 | — |

## 自洽性与质量自检

### 输出前自检表

| 检查项 | 问题 | 通过标准 |
|--------|------|---------|
| 🎯 核 | 5大撤裁事由是否全部预演？ | 5事由均有攻击路径分析，无缺项 |
| 📋 全 | 每个攻击路径是否有事实支撑？ | 每条攻击路径标注了对应裁决书段落或案卷记录 |
| ✅ 精 | 🔴高风险项是否有加固建议？ | 每条🔴都有可执行加固建议+跟踪状态 |
| ⚠️ 疑 | 是否区分了AI判断与人类判断？ | 人/AI判断不一致处已双记录 |
| 🔗 序 | 加固状态可追踪？ | 每条加固建议标注了待处理/已完成/不处理 |

### 🛑 强制回退规则

- **任一❌ → 退回对应步骤补齐**
- **连续2次❌同一检查项** → 建议先走 校核.skill 完成裁决书质检后再来
- **CHECKPOINT 用户中止** → 保存当前进度
- **全部✅ → 输出防讼报告，出裁决前确认所有🔴加固项已处理**
